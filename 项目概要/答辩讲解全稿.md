# Encryption-Traffic-Detection 项目答辩讲解全稿

> 适用场景：课程答辩 / 项目汇报 / 被老师追问时的技术解释

> 项目关键词：加密流量检测、会话级建模、Unknown 检测、可解释分析、实时可视化

---

## 1. 项目一句话介绍（开场30秒）

本项目是一个“面向加密流量的实时检测与研判系统”。
核心思路是：
1. 不依赖明文内容，而是直接用**原始字节序列**建模。
2. 先做**已知类别分类**，再做**未知流量检测（Open-set）**。
3. 把推理结果、可解释证据和 AI 研判打通到前端看板，实现“检测-告警-解释-处置建议”的闭环。

---

## 2. 为什么做这个项目（问题背景）

### 2.1 传统痛点

1. 流量加密后，端口和浅层统计特征区分能力下降。
2. 仅做封闭集分类会把“新型流量”硬分到已知类，误导安全人员。
3. 单纯输出类别标签可解释性弱，不利于答辩和真实运维。

### 2.2 本项目的目标

1. 在加密场景下保持有效分类能力。
2. 对“超出训练分布”的样本给出 suspected / confirmed unknown 分级。
3. 提供包级、字节级解释 + AI 文本化研判，支持人工决策。

---

## 3. 系统总体架构（端到端链路）

可以按“7段流水线”讲：

1. **数据构建层**：`costSensitive/session_data.py`
   - 从 ISCX pcap/pcapng 提取会话样本。
   - 统一成 `[num_packets, packet_len]`（默认 `[12, 256]`）的字节矩阵。

2. **训练层**：`costSensitive/train_byte_session.py`
   - 训练 ByteSessionClassifier。
   - 输出 `byte_session_classifier.pth`、训练日志、训练报告。

3. **离线评估层**：`costSensitive/inference.py`
   - 导出预测 CSV、会话 embedding、评估报告。

4. **Unknown 检测构建层**：`costSensitive/build_centroid_detector.py`
   - 用训练集 embedding 建每类中心与阈值，保存 `centroid_detector.json`。

5. **实时推理层**：`costSensitive/main_realtime.py` + `realtime/single_pipeline.py`
   - 抓包 -> 会话聚合 -> 批量推理 -> unknown 分级 -> 生成解释字段。
   - 写入 `realtime_predictions.csv/jsonl`。

6. **后端服务层**：`services/backend_engine.py` + `services/api_server.py`
   - backend_engine 负责 JSONL 增量入库 SQLite。
   - FastAPI 提供概览、告警、XAI、聚类、AI 洞察和 WebSocket。

7. **前端展示层**：`web-app/`
   - Vue3 + Pinia + Element Plus + ECharts。
   - 页面：Overview / Alerts / XAI / AI Insights。

## 4. 数据与样本构建（老师常问）

### 4.1 数据来源与目录

- 数据位于 `costSensitive/dataset/iscx/`，包含 NonVPN 与 VPN 多类业务流量。
- 构建结果在 `costSensitive/processed_full/sessions/`：
  - `manifest.csv`
  - `label_map.json`
  - `samples/*.npz`

### 4.2 样本定义

每个样本是“会话级”：
1. 一个双向 flow（按五元组规范化）聚合出的前 `num_packets` 个包。
2. 每个包截断/补齐到 `packet_len` 字节。
3. 得到矩阵：`[12, 256]`（默认）。

### 4.3 标签映射策略

`session_data.py` 里维护了 NonVPN/VPN 前缀映射规则，并做了文件名归一化（例如去尾缀、最长前缀匹配）。
这一点很关键：避免因为抓包文件命名差异导致样本被错误跳过。

## 5. 模型设计（重点讲清楚）

模型定义在 `costSensitive/realtime/model_def.py`，可分 3 层：

### 5.1 包级编码器 PacketEncoder

输入：单个包字节序列 `[packet_len]`。

结构：
1. Byte Embedding：`257` 个 token（`0-255` + `PAD=256`）。
2. 1D Conv 三层：
   - Conv1: `emb -> 64`，kernel=5
   - Conv2: `64 -> 128`，kernel=5
   - Conv3: `128 -> 128`，kernel=7
3. AdaptiveMaxPool1d(1) + Linear 投影

输出：每个包一个定长向量（默认 128 维）。

### 5.2 会话级编码器 SessionEncoder

输入：`[B, num_packets, packet_feat_dim]` + `packet_mask`。

结构：
1. BiGRU（双向，hidden=128）。
2. 使用 `pack_padded_sequence` 处理真实包长度。
3. 对有效时间步做 masked mean pooling。

输出：会话特征（`2*hidden=256`）。

### 5.3 分类头

1. `Linear(256->128) + ReLU + Dropout`
2. `Linear(128->num_classes)`

输出：
1. 分类 logits
2. 会话 embedding（用于 Unknown 检测和聚类）

### 5.4 训练策略

在 `train_byte_session.py`：
1. 损失：带类权重的 CrossEntropy（缓解类不平衡）。
2. 优化器：AdamW。
3. 学习率调度：ReduceLROnPlateau。
4. 早停：按 test_acc 的 patience 触发。
5. 输出：log、best model、per-class/macro 指标报告。

## 6. Unknown 检测机制（创新点之一）

### 6.1 为什么需要

封闭集分类只能“在已知类里二选一/多选一”，遇到新分布会误判。项目通过“分类 + 距离判别”引入 open-set 能力。

### 6.2 构建方法

`build_centroid_detector.py`：
1. 用训练集 embedding 按类别计算 centroid。
2. 计算每类样本到中心的距离分布。
3. 取距离分位数（默认 0.95）作为该类阈值。

### 6.3 在线判别规则

`unknown_detector.py` 核心分数：

$$
\text{anomaly\_score} = \frac{\text{centroid\_distance}}{\text{centroid\_threshold}}
$$

默认阈值：
1. `score < 1.0` -> known
2. `1.0 <= score < 1.5` -> suspected
3. `score >= 1.5` -> confirmed_unknown

输出状态：
- `unknown_level`（0/1/2）
- `unknown_state`（known/suspected/confirmed_unknown）
- `is_suspected`、`is_unknown`

## 7. 实时推理与解释（工程亮点）

### 7.1 实时流程

`main_realtime.py` 调 `single_pipeline.py`：
1. Scapy 抓包。
2. FlowAssembler 组会话。
3. 满足触发条件后进入批推理。
4. 得到分类 + unknown 判别 + 告警等级。
5. 生成解释字段并写 CSV/JSONL。

### 7.2 告警等级策略

根据 unknown 分级：
1. known -> low
2. suspected -> medium
3. confirmed unknown -> high

### 7.3 XAI 字段

在 medium/high 告警上做解释：
1. **包级贡献**：occlusion（遮挡一个包看置信下降）。
2. **字节级热力图**：conv3 的 Grad-CAM 风格热力。

说明：项目里特意避免了 inplace ReLU 对 Grad-CAM hook 的不稳定影响，这是一个真实工程坑位修复点。

## 8. 后端架构（FastAPI + SQLite）

### 8.1 两个后端角色

1. `backend_engine.py`
   - 监听 `realtime_predictions.jsonl`。
   - 逐行转换后写入 `netguard_logs.db` 的 `traffic_alerts` 表。
   - 按窗口触发 AI 研判并写 `ai_insights` 表。

2. `api_server.py`
   - 提供 REST API 和 WebSocket。
   - 支持地理解析缓存、unknown 聚类查询、XAI解释生成/缓存。

### 8.2 数据库核心表

1. `traffic_alerts`
   - 基础字段：时间、源/目的IP、协议、类别、置信度、风险分、告警级别
   - 扩展字段：centroid_distance、centroid_threshold、explain_reason、evidence_json
   - XAI字段：packet_contrib_json、byte_heatmap_json

2. `ai_insights`
   - 存储 AI 文本洞察（alert/behavior 两类）

### 8.3 API 组织（前端一一对应）

1. 概览：`/api/overview`
2. 告警：`/api/alerts`

4. unknown 聚类：`/api/unknown/clusters/*`
5. XAI：`/api/xai/samples`、`/api/xai/detail/{id}`、`/api/xai/explain/{id}`
6. AI洞察：`/api/ai/insights`
7. 模型对比：`/api/model/metrics`

WebSocket：
1. `/ws/overview`
2. `/ws/alerts`
3. `/ws/ai`

## 13. 你必须会回答的“高频追问”

### Q1：为什么不用传统统计特征+RF就行？

A：
1. 加密场景下传统特征可分性下降。
2. 本项目直接从字节序列学表示，保留细粒度模式。
3. `model_metrics` 里还做了 RF/CNN/NetGuard 对比，能体现性能与 OOD 能力优势。

### Q2：Unknown 检测为什么可信？

A：
1. 不是拍脑袋阈值，而是训练 embedding 的类内距离分位数建模。
2. 实时决策用 distance/threshold 比值，跨类别可比。
3. 有三级状态，避免“一刀切”误报。

### Q3：XAI 的价值是什么？

A：
1. 告诉你“哪个包”影响最大（包级）。
2. 告诉你“哪个字节区域”可疑（字节级）。
3. 再结合 AI 文本解读和建议，降低分析门槛。

### Q4：实时性怎么保证？

A：
1. 会话组装 + batch 推理，平衡吞吐和时延。
2. 推理结果先写 JSONL，再由后台异步入库，解耦链路。
3. 前端 WebSocket 增量更新，不需要频繁全量轮询。

### Q5：有哪些工程容错？

A：
1. API/前端都做了 payload 归一化与容错解析。
2. AI 模块不可用时自动回退规则解释，不阻断主流程。
3. 地理解析有本地缓存，降低在线接口依赖和延迟。

## 14. 项目局限与改进方向（诚实但不减分）

### 14.1 当前局限

1. unknown 检测目前是 centroid + 阈值，边界表达能力有限。
2. 解释模块对极短会话的稳定性会下降。
3. 地理定位依赖公网解析服务，私网 IP 无法定位。
4. 目前主要是单机 SQLite 方案，不是分布式高可用架构。

### 14.2 可落地改进

1. 引入更强 OOD 方法（energy-based、Mahalanobis、one-class head）。
2. 增加在线增量学习和阈值自适应。
3. 把 SQLite 升级到 PostgreSQL + 队列中间件。
4. XAI 增加对比解释（同类正常样本 vs 告警样本）。

## 17. 快速术语表（防止现场卡壳）

1. **会话（Session）**：按 flow 聚合的一段连续网络通信。
2. **Embedding**：模型学习出的低维语义向量表示。
3. **Open-set / OOD**：测试样本不属于训练已知类别的情形。
4. **Centroid Detector**：以类中心距离判定未知风险的方法。
5. **Occlusion**：遮挡输入局部，观察输出变化以评估贡献。
6. **Grad-CAM（此处是1D变体）**：基于梯度与激活生成热力解释。
